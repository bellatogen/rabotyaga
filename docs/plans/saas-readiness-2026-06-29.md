# SaaS-готовность «Работяга» — план

_Создан: 2026-06-29. Статус: черновик. Владелец: —_

Цель: превратить однотенантное приложение одного бара в безопасный, отказоустойчивый, масштабируемый SaaS, где **каждый компонент настраивается данными из UI без доступа к коду**.

Легенда приоритетов: **P0** — блокер до первых платящих клиентов; **P1** — до горизонтального масштабирования; **P2** — зрелость.

---

## 0. Контекст (что уже крепко)

- Персистенция: PG-first + dirty-tracking + coalescing + сериализация флашей + ретрай PG + файловый резерв (`server.js` `saveData/flushToPG/hydrateFromPG`).
- Auth: bcrypt + httpOnly-cookie + JWT, rate-limit на login (10/15мин) и has-password (20/мин).
- Пуши: единый чокпоинт `sendPush` с ретраем/логом, идемпотентный сид моделей после hydrate.

Хрупкое звено — **контракт фронт↔сервер** (`api.js` молчаливые fallback'и) и **тотальный хардкод одного бара**.

## 1. Мультитенантность (P0) — сейчас ОТСУТСТВУЕТ

- [ ] Таблица `tenants` (id, slug/поддомен, план, статус, created_at).
- [ ] `tenant_id` в PK/индексы всех таблиц: `kv_store`, `employee_bindings`, `push_log`, `push_schedule`, `tasks`, `task_completion`.
- [ ] Резолв тенанта: поддомен (`bar1.rabotyaga55.ru`) или claim в JWT. Middleware кладёт `req.tenantId`.
- [ ] Row-Level Security в Postgres (`SET app.tenant_id`), чтобы утечка между тенантами была невозможна на уровне БД, а не только кода.
- [ ] Онбординг: создать организацию → первого админа → засеять дефолты (роли, seed-задачи, нормы).
- [ ] Убрать хардкод-аккаунты `developer`/`manager` как строки → таблица `users(tenant_id, account, role)`.

## 2. Безопасность (P0)

- [ ] 🔴 **`/api/bind` подделка.** telegramId берётся из тела → перехват чужих пушей/выдача за менеджера. Брать id только из подписанного Telegram `initData` (validate hash по bot token); bind только своего аккаунта. **← начать отсюда.**
- [ ] 🔴 **XSS в пушах.** `parse_mode:'HTML'` + неэкранированные имена/задачи (`sender.js substVars/renderPush`). Экранировать все подстановки. **← начать отсюда.**
- [ ] **KV-запись без схемы.** Любой авторизованный пишет произвольные ключи/значения. → Валидация по схеме (zod/ajv) на ключ; whitelist ключей вместо blacklist; лимит размера значения.
- [ ] **CSRF.** Cookie-auth → `SameSite=Strict/Lax` + CSRF-токен на всех мутациях.
- [ ] **Секреты per-tenant.** iiko/mozg/telegram-креды из общего `.env` → шифрованное хранилище на тенант (`pgcrypto`/KMS).
- [ ] **Аудит.** `audit_log(tenant, actor, action, key, before, after, ts)` — серверный, не клиентский `events_log`.
- [ ] **Привилегии.** Пересмотреть `MANAGER_ONLY_KV` → единый ролевой ACL, проверяемый по `req.user.role`, а не по строке аккаунта.

## 3. Hardening / анти-DDoS (P1)

- [ ] Глобальный rate-limit на весь `/api` (per-IP + per-tenant + per-account), не только login.
- [ ] `express.json({limit:'100kb'})`, серверные таймауты, `helmet` (security-заголовки).
- [ ] WAF/прокси перед приложением (Cloudflare/Timeweb) — L7, бот-фильтр, гео.
- [ ] Идемпотентность мутаций (Idempotency-Key) — защита от дабл-сабмита/ретраев.
- [ ] Backpressure: лимит конкурентных записей на тенант; защита `saveData`-шторма.

## 4. Конфиг-из-UI (P0 для универсальности)

Захардкожено (`// SERVER:`): `HOLIDAYS`, `EMBEDDED_EVENTS`, `staffNorm/staffCheck`, `DEFAULT_MEMBERS`, `HOUR_NORMS`, `ROLES`/perms, seed-задачи, расписание.

- [ ] Каждое → редактируемый стор: **админ-CRUD + валидация + версионирование + дефолтный сид на тенант**.
- [ ] Конструктор ролей/прав из UI (вместо `ROLES` в коде).
- [ ] Редактор правил укомплектованности смены (числа/дни/события).
- [ ] Управление интеграциями (ключи iiko/mozg, маппинг кранов) из UI + кнопка «проверить соединение».
- [ ] Праздники/события — редактируемый справочник (сейчас 2026 захардкожен).

## 5. Масштабируемость / модель данных (P1)

- [ ] Выносить горячее из KV-блобов в реляционные таблицы (tasks/completion/bindings — адаптер частично готов).
- [ ] **Stateless-инстансы:** стейт только в PG/Redis (сейчас в памяти процесса → один инстанс).
- [ ] **Live-обновления через WebSocket/SSE** вместо поллинга (поллинг = нагрузка ×N клиентов + грабли rate-limit, см. инцидент has-password).
- [ ] Redis pub/sub для рассылки изменений между инстансами.
- [ ] PgBouncer при росте числа тенантов.

## 6. Наблюдаемость (P1)

- [ ] Структурные логи (pino) + correlation-id.
- [ ] Метрики (Prometheus) + дашборд; алерты: `pg:false` дольше N мин, рост 5xx/429, провал пушей.
- [ ] Разделить liveness/readiness; алерт на затяжной PG-ретрай (сейчас молчит для оператора).

## 7. Надёжность контракта фронт↔сервер (P0, из прошлого ревью)

- [ ] `sv()` возвращает `{ok,status}`; UI показывает ошибку записи, не фантом-успех; ретрай критичных ключей.
- [ ] `ld()` отдаёт дискриминатор (нет-данных vs нет-доступа); pre-auth экраны — только публичные эндпоинты (паттерн `/api/roster`).
- [ ] Атомарный серверный эндпоинт управления составом (members+profiles+archive за одним гейтом).
- [ ] Никаких per-item fetch-циклов; только batch; на 429 — backoff.
- [ ] `sender.js logPush` — `text`-аргумент получает `type` (баг логирования), починить.

## Тест-матрица

**Безопасность**
- bind-спуфинг отвергнут (чужой telegramId/имя).
- XSS в пушах экранируется.
- KV-валидация режет мусор/oversize.
- CSRF без токена → 403.
- Изоляция тенантов: юзер A не читает/пишет данные B (включая на уровне RLS).
- Границы rate-limit (login, глобальный, per-tenant).

**Отказы**
- PG-down на старте → сервер жив на файле; reconnect → flush мирит удаления (сценарий C1).
- `ld/sv` при 401/403/429/500/network → нет фантом-успеха; 429-backoff.
- Coalescing при конкурентных записях.

**Конфиг-из-UI**
- Каждый бывший хардкод редактируется из UI и переживает рестарт.
- Валидация отвергает кривые значения.

**Нагрузка**
- k6/artillery: N тенантов × M юзеров, шторм PUT — latency, coalescing, деградация под лимитами.

**Контракт**
- Схемы ответов API (контракт-тесты) — изменение бэка не ломает фронт молча.

## Порядок спринтов

1. **Спринт 1 — P0-безопасность:** bind-спуфинг, XSS-пуши, KV-валидация, CSRF, security-заголовки. + тесты красным на дыры.
2. **Спринт 2 — Тест-харнесс + контракт фронт↔API:** фикс `api.js` (`ld/sv` статусы), атомарный roster, отказо-тесты.
3. **Спринт 3 — Мультитенант-скелет:** `tenant_id` + RLS + резолв + онбординг (переделывать позже дорого).
4. **Спринт 4 — Конфиг-из-UI:** по компонентам (роли, нормы, праздники, интеграции).
5. **Спринт 5 — Масштаб:** stateless + Redis pub/sub + WebSocket + наблюдаемость + нагрузочные тесты.
