## Задача (3 части)

### 1. Бэк — мульти-маппинг (`rabotyaga-bot/`)
- **Модель `src/taps/model.js`**: `iikoProductId` теперь `string | string[] | null`. Обновить комментарий. Хелпер `tapIikoNames(tap)` → нормализует в массив непустых строк (строку оборачивает в `[...]`, null/'' → `[]`). Экспортировать.
- **Роут `src/api/taps.js`**:
  - `POST`/`PUT` валидация: `iikoProductId` принимает строку, массив строк или null. Массив — отфильтровать пустые/не-строки, тримить.
  - `POST /refresh-sales`: для каждого крана с непустым набором имён (через `tapIikoNames`) — **суммировать** `counts` по ВСЕМ его именам из `getDishSalesCounts(from,to)` (окно 30 дней), записать в `salesPerMonth`. Краны без имён не трогать. Имя, которого нет в counts → 0 (не падать). Вернуть отчёт: сколько кранов обновлено, какие имена не нашлись (`unmatched`) — пригодится для отладки.
  - computeTap НЕ трогать.

### 2. Фронт `frontend/src/pages/TapsTab.jsx`
- Бейдж 🔗IIKO если набор имён непуст (учесть и строку, и массив — defensive).
- Поле `salesPerMonth` заблокировать если набор имён непуст (продажи из iiko).
- Редактор маппинга в деталях: список имён (textarea — одно имя на строку, или чип-список). Сохранение → массив строк. Подсказка: «Точные DishName из iiko, по одному на строку».
- Не падать если `iikoProductId` пришёл строкой (старые данные) — нормализовать при отображении.
- `tapCompute.js` НЕ трогать (формулы те же).

### 3. Проставление на проде + refresh (скрипт через PG-adapter)
Данные прода в PostgreSQL (kv_store, ключ `taps:v1`). Писать в файл бесполезно (hydrate из PG затрёт). Нужен скрипт `scripts/seed-tap-iiko-mapping.js`:
- Через `db/adapter` (PG): `kvGet('taps:v1')` → распарсить → проставить `iikoProductId` массивы по таблице ниже (матчинг по `id` крана t1..t21) → `kvSet('taps:v1', ...)`.
- Затем (опционально в том же скрипте или отдельным дёрганьем `/refresh-sales`): подтянуть продажи 30 дней через `iiko.getDishSalesCounts` и записать `salesPerMonth`. Если проще — после деплоя дёрнуть HTTP `/api/taps/refresh-sales` с прода (но там requireAuth — нужен токен; скрипт через adapter надёжнее).
- Запуск на проде: `docker exec rabotyaga-bot node scripts/seed-tap-iiko-mapping.js`.
- ВАЖНО: после kvSet проверить что записалось; сервер при следующем чтении возьмёт из PG. Если сервер уже запущен и держит data.kv в памяти — апдейт в PG не виден до рестарта/hydrate. Поэтому: либо рестартовать контейнер после скрипта (`docker restart rabotyaga-bot`), либо скрипт дёргает работающий сервер через HTTP. Простейшее: скрипт пишет в PG → `docker restart rabotyaga-bot` → затем HTTP refresh или повторный скрипт для продаж. Выбери надёжный порядок и опиши.
