# Admin Tab for Developer Profile: Plan

## Goal

Add an "Администратор" tab to the frontend (App.jsx) accessible only to the `developer` and `manager` roles, wiring it to the existing backend admin API (`/api/admin/*`) for managing push templates and push settings.

## Background

### Admin API (backend)
`rabotyaga-bot/src/api/admin.js` — mounted at `/api/admin` (server.js:18). Five endpoints:
- `GET /api/admin/push-settings` — all push settings from `data.json` (`pushSettings` key)
- `GET /api/admin/templates/:userId` — per-user push templates
- `POST /api/admin/templates/:userId` — merge-update per-user push templates
- `GET /api/admin/default-templates` — hardcoded global templates
- `POST /api/admin/default-templates` — merge-update `data.defaultTemplates`

**No auth middleware on any route** (admin.js:17–80). `data.adminUsers` field exists in server.js:31 but is dead — nothing reads it.

### Developer role (frontend)
- Not in `ROLES` object (App.jsx:143) — handled as a hardcoded special case
- `hasPerm(who, profiles, perm)`: if `who === "developer"` → unconditional `true` (App.jsx:236)
- `isManager = who === "manager" || who === "developer"` (App.jsx:538)
- Authenticated via `auth:v1` KV store password — same flow as other users, but first login sets the password (App.jsx:659–669)
- Developer-only feature: ACL toggle "Управляющий может видеть пароли" (App.jsx:1318), gated by `isDeveloper` flag (App.jsx:539)

### Frontend nav structure
Tabs rendered conditionally in the nav bar (App.jsx:761–763):
- Tasks tab: `hasPerm(...,"view_all_tasks") || hasPerm(...,"view_own_tasks")`
- Schedule tab: `hasPerm(...,"view_schedule")`
- Team tab: `canTeam || canStats`

No admin tab exists anywhere in App.jsx. Navigation is tab-state (`tab` in root `App` component) — no router.

### Push stats API
`GET /api/push/stats` (src/api/push.js) — reads `push-log.json`, returns total/sent/failed/skipped + per-user breakdown. Could be surfaced in admin UI alongside template management.

## Approach

Добавить вкладку "Администратор" в `App.jsx`, доступную через `isManager` (`manager` + `developer`). Три подраздела: дефолтные шаблоны пушей, шаблоны по пользователям, статистика пушей. Всё через существующий `/api/admin/*` + `/api/push/stats` — KV store не задействован.

**Одновременно** нужен баг-фикс в бэкенде: `GET /api/admin/default-templates` игнорирует `data.defaultTemplates` и возвращает захардкоженные строки, из-за чего POST-сохранения теряются при перезагрузке.

### Паттерн компонента

`AdminTab` строится по образцу `TeamHubTab` (App.jsx:~1335):
- локальный `sub` state (`"templates" | "users" | "stats"`)
- chip-row переключатель подразделов
- три секции: `DefaultTemplatesSection`, `PerUserTemplatesSection`, `PushStatsSection`

### Загрузка данных

```
mount: Promise.all([
  GET /api/admin/push-settings,
  GET /api/admin/default-templates,
  GET /api/admin/bindings,        ← нужен для маппинга Telegram ID → имя в sub-tab B
])
+ отдельно GET /api/push/stats (ошибка не блокирует остальное)
```

Всё в локальном state, без `usePersist`, без localStorage. `AdminTab` не тянет коллбэки в App-состояние — полностью самодостаточен.

### ⚠ Race condition: два писателя, один `data.json`

`admin.js` пишет напрямую через `fs.writeFileSync` (admin.js:12). `server.js` держит long-lived in-memory `data` объект и флашит его по debounce-таймеру (server.js:45–50). Если admin POST пишет на диск, а следующий debounce флаш server.js перезаписывает из stale in-memory состояния → тихая потеря данных.

**Решение (Item 1):** после записи в `admin.js` синхронизировать in-memory состояние server.js — либо через экспортированный `saveData` helper, либо обновив общий `data` объект. Конкретный механизм — выбор имплементатора; главное — не оставлять `fs.writeFileSync` без синхронизации.

### Новые CSS-классы

Не нужны — все стили уже есть (`.tabs/.tab.on`, `.field`, `.btn-p`, `.info-box`, `.sec`, `.stat-c`, `.grid2`, `.alert`).

### Безопасность

Бэкенд-роуты не защищены — `isManager` только на фронте. Добавить `// SECURITY: нет backend-авторизации на /api/admin/*` рядом с fetch-вызовами.

## Work Items

### Item 1 — Fix backend: persistence + GET /default-templates
**Goal:** (1) `GET /api/admin/default-templates` мёрджит `data.defaultTemplates` поверх хардкода. (2) POST-хендлеры в `admin.js` синхронизируют in-memory `data` объект `server.js` после записи — устраняет race condition с debounce-флашом.
**Done when:** `POST /api/admin/default-templates` → reload страницы → GET возвращает сохранённый текст без перезапуска сервера. Старый `data.json` без `defaultTemplates` ведёт себя как раньше (`{}` мёрдж = хардкод).
**Key files:** `rabotyaga-bot/src/api/admin.js:~8,12,54` — `loadData/saveData` + GET /default-templates handler; `rabotyaga-bot/server.js:45–50` — debounce flush механизм
**Dependencies:** Предшествует Items 2–5 (без фикса template-сохранения — sub-tab A нетестируем).
**Size:** S (15–25 строк)

### Item 2 — Tab entry + AdminTab skeleton
**Goal:** Вкладка "Администратор" появляется в nav-баре только для `isManager`; компонент монтируется, грузит данные, рендерит loading/error/chip-switcher.
**Done when:** `manager` и `developer` видят вкладку и chip-переключатель; остальные роли — нет; ошибки сети видны в alert.
**Key files:** `frontend/src/App.jsx:~762` (tabs array), `~808` (render block), конец файла (новый компонент)
**Dependencies:** Item 1 (нужен рабочий API для тестирования монтирования)
**Size:** S (~65–80 строк)

### Item 3 — DefaultTemplatesSection (sub-tab A)
**Goal:** Просмотр и редактирование шаблонов пушей по умолчанию.
**Done when:** Правки сохраняются через POST и живут после перезагрузки без рестарта.
**Key files:** `frontend/src/App.jsx` — внутри AdminTab
**Notes:** API хардкодит 2 ключа (`dayBeforeShift`, `closeShiftReminder`). Третий тип (`personalTasks`, 09:00 в scheduler.js) ключа в API нет — не добавлять без расширения backend; UI показывает только существующие ключи.
**Dependencies:** Item 2
**Size:** S (~50 строк)

### Item 4 — PerUserTemplatesSection (sub-tab B)
**Goal:** Выбор пользователя из `pushSettings` и редактирование его шаблонов.
**Done when:** `<select>` показывает имена (не raw ID); правки сохраняются через POST.
**Key files:** `frontend/src/App.jsx` — внутри AdminTab
**Notes:** `pushSettings` ключи = Telegram ID (целые числа). `data.bindings` — `{name: telegramId}` (server.js:211) — нужно инвертировать при монтировании для маппинга ID→имя в dropdown. Если `pushSettings` пустой (нет bind'ов) — секция показывает пустой select с hint. **Реализовать GET /api/admin/bindings на бэкенде** или прокинуть через существующий `/api/kv/bindings` если такой ключ есть.
**Dependencies:** Item 2
**Size:** S (~65 строк)

### Item 5 — PushStatsSection (sub-tab C)
**Goal:** Read-only статистика отправок пушей из `/api/push/stats`.
**Done when:** Показываются total/sent/failed/skipped + список byUser с именами (через тот же bindings-маппинг из Item 4).
**Key files:** `frontend/src/App.jsx` — внутри AdminTab
**Dependencies:** Item 2; bindings-маппинг из Item 4 (shared).
**Size:** S (~40 строк)

## Open Questions

*Все три вопроса закрыты:*
- **Доступ**: `isManager` (manager + developer оба видят вкладку)
- **Backend auth**: не добавляем — отложено до реализации bcrypt+session; добавить `// SECURITY:` комментарий в фетч-вызовах
- **Объём**: все три подраздела (Items 1–6)

*Остаётся открытым:*
