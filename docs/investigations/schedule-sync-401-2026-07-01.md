# Расследование: 401 от Google Sheets gviz при синке расписания (01.07.2026)

Всё расследование ниже (вероятностная блокировка IP, флапающие 401) проводилось на
**чужом ID таблицы** — захардкоженном дефолте в коде (`1qu2vBtdSboXhFUCvCjs9XZJOqWeBfo-0`),
который совпадает с публичной таблицей (без пароля/логина). НО фактически прод в `.env`
использовал `SCHEDULE_SHEET_ID=1kcoTm9RyUknJn2brVV7RpcEVWIm1Azmf` — **другую, приватную**
таблицу, требующую логина в Google (редиректит на `accounts.google.com`, стабильный 401
без единого успеха). `process.env.SCHEDULE_SHEET_ID || 'дефолт'` переопределял дефолт кода
этим левым/устаревшим ID — отсюда и постоянный (не флапающий!) сбой именно `scheduleSync`,
пока `revenueSync` (чей `.env`-ID совпадал с дефолтом кода) работал исправно. Настоящая
причина — банальная рассинхронизация конфига, а не мистика на стороне Google. Данные,
восстановленные вручную ниже (см. «Что сделано»), брались из ПУБЛИЧНОЙ таблицы — по факту
совпадающей с реальным расписанием (проверено побайтово при миграции на API v4), так что
импорт корректен. Урок: **при расследовании сначала проверять фактический `.env` на сервере,
а не полагаться на дефолт в коде.**

## Summary (первоначальная версия — сохранена для истории хода расследования)
Автосинк расписания (`scheduleSync.js`) ловил `Sheets HTTP 401` на прод-сервере (Timeweb)
при обращении к анонимному gviz CSV-экспорту Google Sheets. Причина — НЕ анти-бот на
User-Agent и НЕ параллелизм запросов (обе гипотезы проверены и отклонены), а вероятностная
блокировка конкретно с IP прод-сервера, похожая на балансировку между разными фронтендами
Google (один и тот же запрос в одну и ту же секунду — то 200, то 401). Отдельно и независимо:
лист «Август 2026» на момент расследования физически не существовал в исходной Google-таблице
(gviz не возвращает ошибку на несуществующий лист, а молча отдаёт данные другого листа — это
поведение уже было обработано guard'ом в коде).

## Симптомы
- UI: «Ошибка расписания: Июль 2026: Sheets HTTP 401 ...; Август 2026: Sheets HTTP 401 ...»
- Автосинк каждые 12ч и ручная кнопка «Синхронизировать сейчас» — оба ловили 401
- iiko-синк (выручка) и revenueSync (план выручки, другой Google Sheet ID) — работали исправно
- curl/node fetch с локальной машины (не прод-сервера) к тому же Sheet ID — стабильно 200
- curl/node fetch С прод-сервера — иногда 200, иногда 401, флапает за секунды

## Проверенные и отклонённые гипотезы
1. **Анти-бот на User-Agent `node`** — отклонено. node-UA и browser-UA с моей машины дают
   одинаковый 200. Подмена UA в коде (`7b6ee8e`) не влияет на реальную причину.
2. **Параллелизм запросов провоцирует троттлинг** — частично отклонено для обычного режима:
   в normal-режиме `sheetsToFetch()` всего 2 листа, пул конкурентности тут не при чём.
   Актуально только для бэкфилла (12-19 листов) — там ограничение пула оправдано и оставлено.
3. **Публичный доступ к таблице сломан (403/доступ отозван)** — отклонено. Таблица открывается
   в инкогнито без проблем, доступ «по ссылке» цел.
4. **Общая репутация IP дата-центра (весь трафик с сервера блокируется)** — отклонено:
   revenueSync (другой Sheet ID) с того же IP в тот же момент отработал без единой ошибки.

## Подтверждённая причина
Блокировка специфична для (IP сервера × конкретный документ), вероятностная — флапает
за секунды, не по времени (не «холодный старт остывает», а буквально каждый следующий
запрос может попасть на другой фронтенд Google с другим вердиктом). Наши собственные
интенсивные ретраи и ручное тестирование (десятки запросов к одному Sheet ID за 30-40 минут
во время расследования) с высокой вероятностью только продлевали/усугубляли блокировку
именно этого документа — блокировка перестала «мигать» и держалась сплошняком последние
~5 минут активного дебага. **Урок: при 401/429 на gviz — НЕ бомбардировать документ ретраями
и ручными пробниками. Каждый лишний запрос может ухудшать ситуацию.**

## Что сделано (тактически, в этой сессии)
- `scheduleSync.js` / `revenueSync.js`: пул конкурентности (POOL=3) для бэкфилла + джиттер
  в бэкоффе ретраев (коммит `e4f9d9b`) — не устраняет причину, но снижает шанс самосозданного
  залпа при бэкфилле.
- `server.js`: `scheduleSyncWithRetry` теперь ретраит, пока есть `errors`, а не только пока
  `daysUpdated === 0` (тот же коммит).
- Данные за Июль 2026 восстановлены вручную: скачаны CSV с рабочей (не заблокированной)
  машины, распарсены идентичной логикой (`scheduleSync.js`), смержены с текущим
  `schedule:v1`/`events:v1` через `GET/PUT /api/kv` (легитимный whitelisted эндпоинт,
  SEC-4) — без обхода работающего процесса и без потери данных.
- Лист «Август 2026» — не синкался, потому что не существует в исходной таблице. Не код-баг.

## Что сделано (системно, на будущее)
- `src/sync/scheduleParse.js` — вынесена вся логика парсинга CSV в общий модуль, используется
  и автосинком, и ручным импортом (не может «разъехаться» поведение).
- `scripts/manual-schedule-import.js` — воспроизводимый CLI-инструмент вместо ручных
  curl/JWT/ssh-танцев: логинится как обычный менеджер (`/api/auth/login`), тянет лист с любой
  рабочей машины, мержит и заливает через штатный `PUT /api/kv/:key`. Пароль — только через
  `MANAGER_PASSWORD` env.
