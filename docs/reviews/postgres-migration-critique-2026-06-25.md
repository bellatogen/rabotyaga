# Критика плана PG-миграции (postgres-migration-2026-06-25.md)

**Scope:** только 5 запрошенных аспектов. Швы dirty-tracking / Item 5 / порядок развёртывания проверены по реальному коду (`server.js`, `db/adapter.js`, `db/pool.js`).

---

## 1. Топ-3 недоспецифицированных шва

### Шов А — удаление ключей и отвязка binding не доходят до PG (корректность снимок-диффа сломана)
План: «собираем список изменённых ключей (включая удалённые)… для каждого `await adapter.kvSet(key, data.kv[key])`». Но:
- `adapter.js` **не имеет** `kvDelete`. Для удалённого ключа `data.kv[key] === undefined`, а `kvSet` (adapter.js:13) при `typeof !== 'string'` зовёт `JSON.stringify(undefined) → undefined` → в PG пишется `NULL`/literal, строка **не удаляется**. После рестарта удалённый ключ воскресает из PG.
- `DELETE /api/bind/:name` (server.js:393) делает `delete data.bindings[name]`. `adapter.bindEmployee` (adapter.js:33) умеет только upsert; `getBindings` (adapter.js:23) фильтрует `active = true`, но **никто не ставит `active = false`**. Отвязка не доходит до PG → воскресает.

Вывод: dirty-tracking «включая удалённые» нереализуем текущим adapter. Нужны `kvDelete(key)` и `unbindEmployee(name)` (soft `active=false`) — это **отдельный пред-шаг к Item 5**, в плане отсутствует. Имплементатор будет гадать.

### Шов Б — async-запись внутри sync debounce-колбэка: гонка на `lastFlushed`
«Fire-and-forget async с try/catch… после успешного flush обновляем `lastFlushed`». При интенсивных `saveData()` (35 точек, debounce 300мс) второй тик может стартовать, пока `await adapter.kvSet` первого ещё в полёте. Дифф второго считается против **устаревшего** `lastFlushed` → либо двойные записи, либо (при оптимистичном апдейте) потерянные. Не определено: (а) сериализация флашей (mutex/«идёт ли flush»), (б) частичный успех — 2 ключа из 3 записались, 1 упал: обновлять ли `lastFlushed` для двух? План говорит «после успешного flush» как про атомарную операцию, которой нет.

### Шов В — `PG_OK`: одностороннее переключение, нет восстановления
Старт: PG упал → `catch` → fallback на файл, `lastFlushed` пуст, «следующий flush прогреет PG когда поднимется». Но `PG_OK=false` **гейтит** запись (Approach «Старт» п.4) → флаш в PG никогда не вызовется → PG не прогреется никогда. Прямое противоречие внутри одной секции. «Опц. ретрай-проба» — единственный механизм возврата `PG_OK→true` — помечена опциональной, т.е. в Done-критериях Item 5 её нет. Состояние «PG поднялся в рантайме» не покрыто: тихий дрейф data.json vs PG.

**Бонус-шов (порядок старта):** bcrypt-IIFE (server.js:147) синхронно после загрузки дёргает `data.kv['auth:v1']` и `saveData()`. План переводит загрузку на async PG-first, но **не указывает**, как переупорядочить async-загрузку относительно IIFE, инициализации `pushSender` (server.js:166) и монтирования роутеров (167–171), которые требуют заполненный `data`. Без явного `await load()` перед этим блоком — гонка «пустой data на старте».

## 2. Баланс специфичности

**Переспецифицировано (тактика имплементатора):**
- «Почему снимок, а не Proxy» — целый абзац обоснования механики. Достаточно одной строки; выбор структуры — дело имплементатора.
- Двойная запись пуш-логов файл+PG как обязательный «переходный» механизм — это тактическое решение, не требование.

**Недоспецифицировано (нужно в плане):**
- Контракт удаления (Шов А) — критично, отсутствует.
- Сериализация флашей и семантика частичного успеха (Шов Б).
- Условие и точка возврата `PG_OK→true` (Шов В).
- `KV_FORBIDDEN` (server.js:358) существует рядом с `KV_BLACKLIST`, но план упоминает только blacklist — уточнить, пишутся ли forbidden-ключи в PG.

## 3. Противоречия / пропущенные зависимости

- **Item 4 vs Item 5:** Item 4 объявлен «функционально независим, валидируется после Item 5». Это не зависимость кода, а зависимость проверки — Item 4 фактически тривиальный grep+ручной прогон. Стоит **слить в Item 5** (его Done-чеклист уже покрывает сквозной путь). Отдельный коммит избыточен.
- **Миграция ДО первого старта с PG vs Item 1 (`depends_on: service_healthy` + автостарт бота):** Item 6 предписывает `docker compose exec rabotyaga-bot node db/migrate-from-json.js`. Но `exec` требует **уже запущенного** бота, а бот при `docker compose up` стартует вместе с PG → грузится из **пустого** PG → in-memory `data` пуст ещё до миграции. Команда саморазрушительна. Нужен либо `docker compose run --rm` одноразовым контейнером ДО старта бота, либо двухшаговый up (`up -d postgres` → migrate → `up -d rabotyaga-bot`). Это **противоречие между Item 1 и Item 6**, меняющее процедуру.

## 4. Риск переусложнения — что вырезать

- **Item 4** → влить в Item 5.
- **Двойная запись пуш-логов** — выбрать одно (PG как primary, файл оставить только если PG_OK=false), не городить «переходный период» с последующим коммитом-уборкой.
- **Абзац Proxy-vs-снимок** — убрать в одну строку.
- **SSL / push_schedule employee_id-баг** — корректно вынесены за скоуп, не трогать (это правильное упрощение, оставить как есть).

## 5. Вопросы, меняющие порядок реализации

1. **Удаляются ли KV-ключи / отвязываются ли сотрудники в рантайме?** Если да (а `DELETE /api/bind` есть) → `kvDelete`+`unbindEmployee` становятся **пред-item к Item 5**, и снимок-дифф должен трекать множество удалённых ключей. Меняет объём Item 5 и порядок.
2. **`docker compose up` поднимает бота автоматически вместе с PG?** Если да → процедура миграции из Item 6 не работает через `exec`; нужен one-shot-контейнер. Меняет Item 1 (compose) и Item 6.
3. **Должен ли `PG_OK` восстанавливаться без рестарта?** Если да → ретрай-проба обязательна в Item 5 Done; если нет → зафиксировать «PG-recovery только через рестарт» явно, иначе тихий дрейф.
4. **Что считается источником истины при конфликте data.json ≠ PG после fallback-сессии?** Определяет, безопасен ли «прогрев» PG из устаревшего файла или нужен ручной reconcile.

**Итог:** план силён по разведке и API-сохранности, но три шва записи (удаление, гонка флашей, восстановление PG_OK) и противоречие «миграция-до-старта vs автостарт бота» оставят имплементатора гадать в самом рискованном месте — целостности данных. Закрыть вопросы 1–2 до старта реализации.
